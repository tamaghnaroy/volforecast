# AutoVolForecaster Review 7

## Findings

1. The candidate pool still mixes forecasters with different declared target variables, but the benchmark ranks them all against the same RV proxy as if they were directly comparable. The design passes a single `list[BaseForecaster]` into `BenchmarkRunner` and describes them all as "the best volatility models from the `volforecast` library" ([auto_volforecaster.md:7](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/docs/issues/auto_volforecaster.md#L7), [auto_volforecaster.md:140](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/docs/issues/auto_volforecaster.md#L140)). But the repo’s models target different latent objects: GARCH-family models declare `CONDITIONAL_VARIANCE` ([garch.py:255](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/volforecast/models/garch.py#L255)), HAR models declare `INTEGRATED_VARIANCE` ([har.py:142](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/volforecast/models/har.py#L142)), and `HARCJForecaster` declares `CONTINUOUS_VARIATION` ([har.py:392](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/volforecast/models/har.py#L392)). `BenchmarkRunner` ignores those target specs and evaluates every forecast against `realized["RV"]` with the same MSE/QLIKE calculation ([runner.py:277](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/volforecast/benchmark/runner.py#L277), [runner.py:280](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/volforecast/benchmark/runner.py#L280), [runner.py:281](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/volforecast/benchmark/runner.py#L281)). That makes the selection problem scientifically incoherent unless the design first partitions candidates by target or explicitly maps each target to an appropriate proxy.

2. The new GARCH fallback path can still produce `BenchmarkResult` objects that are incompatible with Phase 4’s per-period loss construction. When all regular candidates fail, `BenchmarkRunner` creates `fb_forecasts` full of `NaN`, evaluates metrics only on the `valid_mask`, but still stores the full `fb_forecasts` array and full `proxies` array in the fallback result ([runner.py:313](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/volforecast/benchmark/runner.py#L313), [runner.py:346](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/volforecast/benchmark/runner.py#L346), [runner.py:351](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/volforecast/benchmark/runner.py#L351), [runner.py:365](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/volforecast/benchmark/runner.py#L365), [runner.py:366](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/volforecast/benchmark/runner.py#L366)). Phase 4 then says it will compute per-period losses from stored forecasts and build a `(T, M)` loss matrix for DM/MCS ([auto_volforecaster.md:184](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/docs/issues/auto_volforecaster.md#L184)). Unless the auto-selector strips those invalid periods before constructing losses, the fallback result will inject `NaN`s into DM testing and MCS.

3. The `effective_loss` logic is still internally inconsistent in the model-selection section. Phase 3 says proxy quality determines a single `effective_loss`, and that this effective loss is used for all downstream ranking and selection ([auto_volforecaster.md:164](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/docs/issues/auto_volforecaster.md#L164), [auto_volforecaster.md:166](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/docs/issues/auto_volforecaster.md#L166)). But the fallback rule in Phase 4 still says that if MCS goes empty, the system should keep the top-3 by QLIKE ([auto_volforecaster.md:197](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/docs/issues/auto_volforecaster.md#L197)). In the exact case where proxy quality has already forced `effective_loss = "MSE"`, this would switch the selection criterion back to QLIKE at the last step and make the final decision rule non-deterministic.

## Open Questions

- Is the intended v1 scope to compare only models targeting the same latent object, or should `AutoVolForecaster` maintain separate leaderboards for conditional variance, integrated variance, and continuous variation before any cross-family combination happens?
- For the GARCH fallback result, should invalid OOS periods be dropped before they ever enter `BenchmarkResult`, or should `BenchmarkResult` grow an explicit mask/valid-index field so Phase 4 can construct aligned loss arrays safely?

## Change Summary

The latest design has resolved most of the earlier API-drift issues. The remaining problems are now narrower and more substantive: cross-target comparability is still not handled, the GARCH fallback can leak `NaN` forecasts into Phase 4, and the `effective_loss` rule is contradicted by the final QLIKE-based survivor fallback.
