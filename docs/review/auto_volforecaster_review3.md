# AutoVolForecaster Review 3

## Findings

1. The fail-safe guarantee is still not implemented end to end. The design says failed models are dropped and the ensemble always has at least one member via a GARCH fallback ([auto_volforecaster.md:18](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/docs/issues/auto_volforecaster.md#L18)), but the current `BenchmarkRunner` only excludes failed models and returns whatever remains ([runner.py:193](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/volforecast/benchmark/runner.py#L193), [runner.py:255](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/volforecast/benchmark/runner.py#L255), [runner.py:285](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/volforecast/benchmark/runner.py#L285)). If every candidate fails, `suite.results` is empty and the downstream DM/MCS stages have nothing to operate on. The proposal needs an explicit recovery rule in `AutoVolForecaster.fit()`, not just a statement in the design notes.

2. The realized-measure contract is still too strict for the public API the design advertises. The issue spec accepts a generic optional `realized_measures` dict ([auto_volforecaster.md:56](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/docs/issues/auto_volforecaster.md#L56), [auto_volforecaster.md:273](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/docs/issues/auto_volforecaster.md#L273)) and explicitly includes models such as `HEAVYForecaster` and `RealizedGARCHForecaster` under the realized-data branch ([auto_volforecaster.md:114](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/docs/issues/auto_volforecaster.md#L114), [auto_volforecaster.md:119](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/docs/issues/auto_volforecaster.md#L119), [auto_volforecaster.md:120](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/docs/issues/auto_volforecaster.md#L120)). But `BenchmarkRunner` currently rejects any precomputed realized bundle unless it already contains all six keys `{"RV","BV","CV","JV","RS_pos","RS_neg"}` ([runner.py:156](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/volforecast/benchmark/runner.py#L156), [runner.py:159](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/volforecast/benchmark/runner.py#L159)), even though `HEAVYForecaster` and `RealizedGARCHForecaster` only require `RV` ([heavy.py:106](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/volforecast/models/heavy.py#L106), [realized_garch.py:120](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/volforecast/models/realized_garch.py#L120)). As written, valid user inputs like `{"RV": ...}` will be rejected before model-specific fitting begins.

3. The selector still omits the rough-volatility family even though the repo now contains rough-vol models and Hurst estimation machinery. The profiler uses a Hurst-based long-memory diagnostic and only maps it to `FIGARCHForecaster` when `H > 0.6` ([auto_volforecaster.md:63](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/docs/issues/auto_volforecaster.md#L63), [auto_volforecaster.md:111](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/docs/issues/auto_volforecaster.md#L111)). But the codebase already ships `RoughBergomiForecaster` and `RoughHestonForecaster` ([models/__init__.py:45](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/volforecast/models/__init__.py#L45)), and the rough-vol module explicitly estimates Hurst with the rough-vol regime `H < 0.5` ([rough_vol.py:74](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/volforecast/models/rough_vol.py#L74), [rough_vol.py:100](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/volforecast/models/rough_vol.py#L100), [rough_vol.py:234](C:/Users/tamaghna%20roy/CascadeProjects/windsurf-project-3/volforecast/models/rough_vol.py#L234)). Given the repo’s stated SOTA ambitions, leaving an existing frontier family out of the auto-selector is now a material model-coverage gap rather than a harmless omission.

## Open Questions

- Should `AutoVolForecaster.fit()` accept partial realized inputs and synthesize any missing measures on demand, or should the public API be narrowed to require a full realized-measure bundle whenever `realized_measures` is supplied?
- If rough-vol models stay out of scope for the first implementation, the design should say that explicitly and justify why the Hurst diagnostic only gates FIGARCH rather than rough-vol candidates.

## Change Summary

This version of the design is much closer to the repository than the earlier drafts. The main remaining issues are now narrower but still important: the fail-safe fallback is not actually wired, the realized-data API is stricter in code than in the spec, and the selector still misses the repo's rough-volatility family.

## Implementation Notes

**FIXED** — All 3 findings addressed:

1. **GARCH fallback guarantee** — `BenchmarkRunner.run()` now checks if `suite.results` is empty after the main loop. If all candidates failed, it automatically runs a `GARCHForecaster()` fallback over the full OOS window and appends it as `"GARCH(1,1)-fallback"`. Downstream DM/MCS stages always have at least one result.

2. **Relaxed realized-measure contract** — `precomputed_realized` now only requires `"RV"`. Missing keys (`BV`, `CV`, `JV`, `RS_pos`, `RS_neg`) are derived automatically from RV. Users can pass `{"RV": rv_array}` and the runner will synthesize the rest. Docstring updated.

3. **Rough-vol family in selector** — Added `RoughBergomiForecaster` and `RoughHestonForecaster` under a new "Conditional on roughness" pool (gated by `H < 0.5` and `T >= 500`). Added `has_rough_vol: bool` to `DataProfile`. Hurst diagnostic now covers both long-memory (`H > 0.6`) and roughness (`H < 0.5`) regimes.
